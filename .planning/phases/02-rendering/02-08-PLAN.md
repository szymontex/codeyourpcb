<task type="auto">
  <name>Task 1: Configure getrandom features for WASM</name>
  <files>
    Cargo.toml
    crates/cypcb-render/Cargo.toml
  </files>
  <action>
The getrandom dependency chain requires specific features for WASM:
- getrandom 0.2.x: needs `js` feature
- getrandom 0.3.x: needs `wasm_js` feature

The current approach in cypcb-render/Cargo.toml adds both as direct dependencies with platform-specific cfg, but this doesn't patch the transitive dependencies.

Instead, use Cargo's patch feature at the workspace level:

1. In workspace Cargo.toml, add a `[patch.crates-io]` section OR use `[target.'cfg(target_arch = "wasm32")'.dependencies]` to force the features:

```toml
# Add to workspace Cargo.toml

[target.'cfg(target_arch = "wasm32")'.dependencies]
getrandom = { version = "0.2", features = ["js"] }

# For the 0.3 version, we need to add it as a workspace dependency
# and ensure the feature is enabled
```

2. In cypcb-render/Cargo.toml, replace the current getrandom entries with proper workspace-level feature activation.

The key insight: Features in Cargo are additive and unioned across all dependents. By adding getrandom with `js`/`wasm_js` features at the workspace level for WASM targets, those features will be enabled for all transitive uses.

Alternative approach if the above doesn't work: Use `[patch.crates-io]` to override the getrandom versions with feature-enabled versions.
  </action>

Run: `cargo check -p cypcb-render --target wasm32-unknown-unknown 2>&1 | grep -i "getrandom\|error"`

Should show no getrandom-related errors.
  </verify>

  </done>
