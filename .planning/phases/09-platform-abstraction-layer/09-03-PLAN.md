  <action>
Create a declarative menu model that both desktop (Tauri native menus) and web (HTML menus) can render from the same data structure. This is intentionally data-only - rendering is deferred to Phase 12 (Desktop) and Phase 13 (Web).

1. **src/menu.rs** - Menu data model:
   ```rust
   use serde::{Serialize, Deserialize};

   #[derive(Debug, Clone, Serialize, Deserialize)]
   pub struct MenuBar {
       pub items: Vec<Menu>,
   }

   #[derive(Debug, Clone, Serialize, Deserialize)]
   pub struct Menu {
       pub label: String,
       pub items: Vec<MenuItem>,
   }

   #[derive(Debug, Clone, Serialize, Deserialize)]
   pub enum MenuItem {
       Action {
           id: String,
           label: String,
           shortcut: Option<String>,  // e.g. "Ctrl+S", "Cmd+O"
           enabled: bool,
       },
       Separator,
       Submenu(Menu),
   }

   impl MenuBar {
       pub fn new() -> Self {
           Self { items: Vec::new() }
       }

       pub fn add_menu(mut self, menu: Menu) -> Self {
           self.items.push(menu);
           self
       }
   }

   impl Menu {
       pub fn new(label: impl Into<String>) -> Self {
           Self { label: label.into(), items: Vec::new() }
       }

       pub fn add_item(mut self, item: MenuItem) -> Self {
           self.items.push(item);
           self
       }

       pub fn separator(mut self) -> Self {
           self.items.push(MenuItem::Separator);
           self
       }
   }

   impl MenuItem {
       pub fn action(id: impl Into<String>, label: impl Into<String>) -> Self {
           Self::Action {
               id: id.into(),
               label: label.into(),
               shortcut: None,
               enabled: true,
           }
       }

       pub fn with_shortcut(self, shortcut: impl Into<String>) -> Self {
           match self {
               Self::Action { id, label, enabled, .. } => Self::Action {
                   id, label, shortcut: Some(shortcut.into()), enabled
               },
               other => other,
           }
       }

       pub fn disabled(self) -> Self {
           match self {
               Self::Action { id, label, shortcut, .. } => Self::Action {
                   id, label, shortcut, enabled: false
               },
               other => other,
           }
       }
   }
   ```

   This is a declarative data model, NOT a trait. Research (Pitfall 5) warns against premature Menu trait abstraction since Tauri native menus and HTML menus are fundamentally different paradigms. The data model lets both platforms read the same menu structure and render it natively.

2. **Update src/lib.rs**: Add `pub mod menu;` and `pub use menu::{MenuBar, Menu, MenuItem};`

Add serde to Cargo.toml dependencies if not already there (it's in workspace deps).

1. **src/platform.rs** - Facade aggregating all platform services:
   ```rust
   use crate::error::PlatformError;
   use crate::dialog::Dialog;
   use crate::fs::FileSystem;
   use crate::storage::Storage;

   // On native: concrete types
   #[cfg(native)]
   use crate::fs_impl::NativeFileSystem;
   #[cfg(native)]
   use crate::storage_impl::SqliteStorage;

   // On WASM: concrete types
   #[cfg(wasm)]
   use crate::fs_impl::WebFileSystem;
   #[cfg(wasm)]
   use crate::storage_impl::WebStorage;  // or whatever the web storage struct is named
   ```

   Create the Platform struct with platform-conditional fields:

   ```rust
   #[cfg(native)]
   pub struct Platform {
       fs: NativeFileSystem,
       dialog: Dialog,
       storage: SqliteStorage,
   }

   #[cfg(wasm)]
   pub struct Platform {
       fs: WebFileSystem,
       dialog: Dialog,
       storage: WebStorage,
   }
   ```

   Implement Platform:
   - `Platform::new_native(storage_path: &std::path::Path) -> Result<Self, PlatformError>` (native only): Creates NativeFileSystem, Dialog, SqliteStorage with given path
   - `Platform::new_web(db_name: &str) -> Result<Self, PlatformError>` (WASM only): Creates WebFileSystem, Dialog, WebStorage
   - Accessor methods (both platforms): `fn fs(&self) -> &impl FileSystem`, `fn dialog(&self) -> &Dialog`, `fn storage(&self) -> &impl Storage`
   - Alternative: use `fn fs(&self) -> &dyn FileSystem` if associated types make `impl` difficult. Or return concrete type behind cfg.

   Simpler approach if generics are complex:
   ```rust
   impl Platform {
       #[cfg(native)]
       pub fn fs(&self) -> &NativeFileSystem { &self.fs }
       #[cfg(wasm)]
       pub fn fs(&self) -> &WebFileSystem { &self.fs }
       pub fn dialog(&self) -> &Dialog { &self.dialog }
       #[cfg(native)]
       pub fn storage(&self) -> &SqliteStorage { &self.storage }
       #[cfg(wasm)]
       pub fn storage(&self) -> &WebStorage { &self.storage }
   }
   ```

   Application code calls `platform.fs()`, `platform.dialog()`, `platform.storage()` and gets the right implementation. The concrete type is hidden behind the trait methods.
