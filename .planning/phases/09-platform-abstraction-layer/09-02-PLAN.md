1. **src/storage.rs** - Platform-agnostic trait:
   ```rust
   use crate::error::PlatformError;
   use async_trait::async_trait;

   #[async_trait(?Send)]
   pub trait Storage {
       async fn init(&self) -> Result<(), PlatformError>;
       async fn get(&self, table: &str, key: &str) -> Result<Option<Vec<u8>>, PlatformError>;
       async fn get_string(&self, table: &str, key: &str) -> Result<Option<String>, PlatformError>;
       async fn set(&self, table: &str, key: &str, value: &[u8]) -> Result<(), PlatformError>;
       async fn set_string(&self, table: &str, key: &str, value: &str) -> Result<(), PlatformError>;
       async fn delete(&self, table: &str, key: &str) -> Result<(), PlatformError>;
       async fn list_keys(&self, table: &str) -> Result<Vec<String>, PlatformError>;
   }
   ```

2. **src/storage_native.rs** - SQLite implementation:
   - `pub struct SqliteStorage { conn: rusqlite::Connection }` (Note: rusqlite::Connection is not Send, so wrap in appropriate sync primitive or use synchronous methods since we're using ?Send)
   - Actually: Since we use `#[async_trait(?Send)]`, we can hold non-Send types. But native code may need Send. Solution: Use `rusqlite::Connection` with blocking operations wrapped in the async methods (rusqlite is synchronous). For native, the async methods just do sync work internally - this is fine since Storage is used from application code that manages its own threading.
   - `SqliteStorage::new(path: &std::path::Path) -> Result<Self, PlatformError>`: Open/create SQLite database at path
   - `init()`: Create tables if not exist - use a single `kv_store` table with columns: `table_name TEXT, key TEXT, value BLOB, PRIMARY KEY (table_name, key)`
   - `get()`: `SELECT value FROM kv_store WHERE table_name = ? AND key = ?`
   - `get_string()`: Same as get but convert bytes to String
   - `set()`: `INSERT OR REPLACE INTO kv_store (table_name, key, value) VALUES (?, ?, ?)`
   - `set_string()`: Convert string to bytes, call set
   - `delete()`: `DELETE FROM kv_store WHERE table_name = ? AND key = ?`
   - `list_keys()`: `SELECT key FROM kv_store WHERE table_name = ?`

3. **src/storage_web.rs** - IndexedDB implementation:
   - `pub struct IndexedDbStorage { db_name: String }`
   - Use web-sys IDB bindings (IdbFactory, IdbDatabase, IdbObjectStore, IdbTransaction) via wasm-bindgen
   - Add required web-sys features to Cargo.toml: "IdbFactory", "IdbDatabase", "IdbObjectStore", "IdbTransaction", "IdbRequest", "IdbOpenDbRequest", "IdbKeyRange", "IdbTransactionMode", "IdbObjectStoreParameters"
   - `IndexedDbStorage::new(db_name: &str) -> Self`
   - `init()`: Open IndexedDB database, create object stores for each table on upgrade. Since we don't know tables upfront, use a single object store "kv_store" with compound key path.
   - For all operations: Open transaction, get object store, perform operation, await result via wasm-bindgen-futures
   - **Simpler alternative if web-sys IDB is too complex:** Use localStorage as initial implementation with JSON serialization. Storage trait abstraction means we can swap to IndexedDB later. localStorage is synchronous and simpler. Use `web_sys::window().unwrap().local_storage().unwrap().unwrap()` methods. Prefix keys with `{table}::{key}` pattern.
   - Choose the simpler localStorage approach for v1.1 - IndexedDB can be added when we need >5MB storage (library phase). Document this as intentional simplification.

4. **Update Cargo.toml**: Add web-sys features needed: "Storage" (for localStorage). Add rusqlite to native deps if not already there.

5. **Update src/lib.rs**:
   - Add `pub mod storage;`
   - Add `#[cfg_attr(wasm, path = "storage_web.rs")]` and `#[cfg_attr(native, path = "storage_native.rs")]` for `mod storage_impl;`
   - Re-export: `pub use storage::Storage;`, `pub use storage_impl::*;`
