Implement context-aware autocomplete (per RESEARCH.md Tree-sitter integration pattern):

1. **CompletionContext enum**:
   - ComponentFootprint (after footprint string quote)
   - NetName (in net definition or pin reference)
   - ComponentName (in pin reference before dot)
   - PropertyKey (inside definition block)
   - LayerName (after "layer" keyword)
   - TopLevel (at document root)
   - Unknown

2. **find_completion_context(ast: &SourceFile, offset: usize) -> CompletionContext**:
   - Walk AST to find node containing offset
   - Determine context from node type and position

3. **completion_at_position(doc: &DocumentState, position: Position) -> Vec<CompletionItem>**:
   - Convert position to offset
   - Get context
   - Generate completions based on context

4. **Completion generators**:

   a. **footprint_completions()** -> Vec<CompletionItem>:
      - Get all footprints from FootprintLibrary
      - Include built-in: "0402", "0603", "0805", "SOIC-8", etc.
      - Add KiCad imports if available
      - CompletionItemKind::Class
      - Detail: "SMD 2-pin" etc.

   b. **net_completions(ast: &SourceFile)** -> Vec<CompletionItem>:
      - Extract all net names from AST
      - CompletionItemKind::Variable
      - Detail: "Net (N connections)"

   c. **component_completions(ast: &SourceFile)** -> Vec<CompletionItem>:
      - Extract all refdes from AST
      - CompletionItemKind::Variable
      - Detail: "resistor" etc. (component kind)

   d. **property_completions(context: &CompletionContext)** -> Vec<CompletionItem>:
      - For component: "value", "at", "rotate"
      - For net: "width", "clearance", "current"
      - For board: "size", "layers"
      - CompletionItemKind::Property

   e. **layer_completions()** -> Vec<CompletionItem>:
      - "Top", "Bottom", "Inner1", "Inner2", etc.
      - CompletionItemKind::Enum

   f. **top_level_completions()** -> Vec<CompletionItem>:
      - "board", "component", "net", "footprint", "trace"
      - CompletionItemKind::Keyword
      - Include snippets with placeholders

5. **Update backend.rs**:
   - Implement completion() method on LanguageServer
   - Declare completion capability with trigger characters: ".", " ", "\""

6. **Tests**:
   - Completion in footprint string
   - Completion after component reference
   - Completion at top level

<task type="auto">
  <name>Task 2: Implement Go-to-Definition</name>
  <files>
    crates/cypcb-lsp/src/goto.rs
    crates/cypcb-lsp/src/backend.rs
  </files>
  <action>
Implement go-to-definition for navigation:

1. **goto_definition(doc: &DocumentState, position: Position) -> Option<Location>**:
   - Find node at position
   - Determine what it references
   - Return location of definition

2. **Navigation cases**:

   a. **Pin reference (R1.1)**:
      - From "R1" part -> go to component definition
      - From "1" part -> go to component definition (can't go to pin)

   b. **Net name in pin assignment**:
      - From net name -> go to net definition block

   c. **Footprint reference**:
      - From footprint string -> go to footprint definition (if custom)
      - Built-in footprints have no location

   d. **Component kind (resistor, led)**:
      - No definition to go to (built-in)

3. **find_definition_location(ast: &SourceFile, name: &str, kind: DefinitionKind) -> Option<Span>**:
   - DefinitionKind: Component, Net, Footprint
   - Walk AST to find matching definition
   - Return its span

4. **Span to Location conversion**:
   - Use document URI
   - Convert byte span to line/column Range

5. **Update backend.rs**:
   - Implement definition() method on LanguageServer
   - Declare definition capability

6. **Tests**:
   - Go to component from pin ref
   - Go to net from net name
   - Go to custom footprint
   - Return None for built-ins
  </action>
  <verify>
    cargo test -p cypcb-lsp goto
  </verify>
  <done>
    Pin references navigate to component definition.
    Net names navigate to net definition.
    Custom footprints navigate to footprint definition.
  </done>
</task>
